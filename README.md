## 1. 背景：なぜこの課題を選んだか

タクシーやライドシェアサービスにおいて、配車効率の悪化は事業者・ドライバー・利用者の三者すべてに損失をもたらす。需要がある地域に車両がいなければ機会損失となり、需要のない地域で空車待機すれば燃料と時間が浪費される。

「いつ・どこで・どれくらいの乗車需要が発生するか」を事前に高い精度で予測できれば、

- 車両を需要発生エリアへ**事前配備**できる
- ドライバーの空車時間を削減し、**実車率を向上**させられる
- 利用者の**待ち時間を短縮**できる

といった改善が一気に進む。本プロジェクトは、メーターが作動した日時と乗車位置情報という**どの配車サービスでも必ず保持しているデータ**だけを使い、地理メッシュ × 時間 × 曜日の粒度で需要を予測するモデルを構築するものである。

> 💡 **本プロジェクトの問い**：
> 「気象・祝日・イベントなどの外生変数を一切使わず、過去の乗車履歴だけを使って、Google Colab 無料枠の計算資源で**実用に耐える時空間需要予測**ができるか？」

## 2. データ

- **Source**: [Kaggle - NYC Taxi Trip Duration](https://www.kaggle.com/competitions/nyc-taxi-trip-duration/data)
- **Samples**: 約 145 万件の乗車履歴（2016 年）→ マンハッタン周辺に絞り込み → **161,448 件**のメッシュ集計データ
- **Target**: `demand`（各メッシュ × 時間 × 曜日における乗車回数）
- **取得手順**: Kaggle から zip をダウンロード → Google Drive 経由で Colab に展開（ノートブック冒頭で自動化）

### 前処理サマリ

| 操作                 | 内容                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------ |
| 範囲フィルタ         | 経度 `[-74.05, -73.75]`, 緯度 `[40.60, 40.90]`（マンハッタン周辺、海上の外れ値を除外）     |
| メッシュ化           | 経度・緯度を小数第 2 位で丸め → 約 **1km × 1km の地理グリッド**（31 × 31 = 961 メッシュ）  |
| 派生特徴量           | `hour`（0–23）, `wday`（日–土）, `is_weekend`（Weekend/Weekday）                            |
| ラグ特徴量           | `demand_1h_ago`（リアルタイム配車支援用）, `demand_24h_ago`（前日バッチ計画用）             |
| ベースグリッド完備   | 乗車がなかった `(メッシュ, 時間, 曜日)` セルは `demand = 0` で補完（ゼロ膨張に対応）         |

最終的な学習データ数: **31 (経度) × 31 (緯度) × 24 (時間) × 7 (曜日) = 161,448 件**

## 3. 手法：なぜ BAM (mgcv) を選んだか

### 3.1 モデル選定ロジック

| 選択肢                       | 採用 | 理由                                                                            |
| ---------------------------- | :--: | ------------------------------------------------------------------------------- |
| LightGBM / XGBoost           |  ❌  | 高精度だが「なぜその時間・場所で需要が高いか」の統計的説明が困難                |
| ConvLSTM / Graph Neural Net  |  ❌  | SOTA 級だが学習コストが高く、Colab 無料枠では現実的でない                       |
| Poisson 回帰 (線形)          |  ❌  | 時間帯の周期性や地理的非線形性を表現できない                                    |
| **BAM (Big GAM)**            |  ✅  | 大規模データで動作する GAM、**時空間テンソル積平滑化 + 周期スプライン**が使える |

### 3.2 モデル定式化

実運用シナリオに応じて 2 種類のモデルを設計した。

#### モデル A：リアルタイム配車支援用（1 時間前ラグ）

```r
demand ~
  wday +
  te(lon_grid, lat_grid, hour,
     d  = c(2, 1),
     bs = c("tp", "cc"),
     k  = c(15, 8),
     by = is_weekend) +
  s(demand_1h_ago, bs = "tp", k = 10)
```

- `te(lon, lat, hour)` の**時空間テンソル積平滑化**で、「金曜 22 時のミッドタウンは高需要、日曜午前のダウンタウンは低需要」といった**位置と時間の相互作用**を表現
- `bs = c("tp", "cc")` — 空間軸は薄板スプライン (`tp`)、時間軸は**循環スプライン (`cc`)** を使い、23 時 → 0 時の不連続性を排除
- `by = is_weekend` — 平日／休日でまったく異なる需要パターン（朝の通勤ピーク vs 深夜の繁華街）を**別スプラインで推定**
- `s(demand_1h_ago)` — 直前 1 時間の需要を非線形に取り込むことで、突発的な需要変動への追従性を確保

#### モデル B：前日バッチ計画用（24 時間前ラグ）

```r
demand ~
  wday +
  te(lon_grid, lat_grid, hour,
     d  = c(2, 1),
     bs = c("tp", "cc"),
     k  = c(15, 8),
     by = is_weekend) +
  s(demand_24h_ago, bs = "tp", k = 10)
```

「翌日の各時間帯のシフト計画」を立てる用途を想定。前日（24 時間前）の同時刻需要をラグとして使うことで、リアルタイム入力が無くても予測が可能。

### 3.3 推定上の工夫

| 項目                    | 設定                                  | 理由                                                                 |
| ----------------------- | ------------------------------------- | -------------------------------------------------------------------- |
| 尤度                    | **Negative Binomial** (`family = nb`) | 需要 (カウントデータ) が過分散 — Poisson では分散を過小評価する     |
| 推定アルゴリズム        | `bam(..., discrete = TRUE)`           | 共変量を離散化してメモリ・計算時間を大幅削減                         |
| 並列化                  | `nthreads = 2`                        | Colab 無料枠の 2 コアを最大活用                                      |
| メッシュゼロ需要の扱い  | `demand = 0` で完全グリッドを補完     | 「乗車がなかった ≠ データなし」を明示的に区別                        |

実装: ノートブック [`ride_hailing.ipynb`](ride_hailing.ipynb) 内で完結

## 4. 結果

### 4.1 モデル比較（全 161,448 件で学習）

| モデル                            | 説明変数                              | Deviance Explained |
| --------------------------------- | ------------------------------------- | -----------------: |
| ベースライン（ラグなし）          | 時空間 te + 曜日                      |          **76.0%** |
| **モデル A: 1h ラグ版**           | 上記 + `s(demand_1h_ago)`             |          **90.6%** |
| モデル B: 24h ラグ版              | 上記 + `s(demand_24h_ago)`            |       高 Deviance  |

> 1 時間前ラグの導入だけで **Deviance Explained が 76% → 90.6% へ大幅改善**。
> 短時間の需要慣性（直前に需要が高かった場所は次の 1 時間も高い傾向）の情報量がいかに大きいかを定量的に示せた。

### 4.2 モデル A の `summary()` 主要結果

```
Family: Negative Binomial(0.453)
Link function: log

Approximate significance of smooth terms:
                                                 edf  Ref.df       F  p-value
te(hour,lon_grid,lat_grid):is_weekendWeekday   84.16   91.06  165.33  < 2e-16 ***
te(hour,lon_grid,lat_grid):is_weekendWeekend   58.14   68.18   99.37  < 2e-16 ***
s(demand_1h_ago)                                8.97    9.00 2711.66  < 2e-16 ***

Deviance explained = 90.6%   n = 161,448
```

- **`s(demand_1h_ago)` の F 値が 2,711 と圧倒的に大きく、最も強力な予測因子**
- 平日・休日の時空間平滑項はどちらも有意で、両者を分離する設計の妥当性が確認された
- 曜日の係数（`wdayMon` ～ `wdayFri` が約 −1.4、`wdaySat` ≈ 0）から、**金土日に需要がはっきり高い**ことが解釈可能

### 4.3 説明可能性

GAM/BAM の最大の利点は、**各説明変数の効果を 1 つのスプライン曲線として可視化できる**ことである。本モデルでは次のような知見が統計的に裏付けられた。

- 平日は朝 7–9 時と夕 17–19 時に明確な双峰需要（通勤ピーク）
- 休日は昼から深夜にかけてのなだらかな単峰需要（飲食・娯楽）
- 同じ時間帯でも、ミッドタウン／空港周辺／住宅地で需要レベルが大きく異なる
- 1 時間前需要との関係はほぼ単調増加だが、**極端な高需要では飽和的にカーブが寝る**（道路供給律速の示唆）

## 5. 考察：単純な数値以上に重要な "モデル設計の含意"

### 5.1 「Deviance 90.6%」の正しい読み方

ラグなしモデル (76.0%) → 1h ラグ版 (90.6%) で精度が 14 ポイントも上がっているが、これは「**直前の需要を見れば直後の需要はかなり当てられる**」という時系列データの当たり前の性質を反映しているに過ぎない。

つまり 90.6% という数字は、

| 言えること                                              | 言えないこと                                                 |
| ------------------------------------------------------- | ------------------------------------------------------------ |
| 「直前 1 時間の需要が観測できる状況」での需要追従性能   | 「何の情報もない状態から将来の需要を予測する」性能           |
| **リアルタイム配車支援**としての実用性                  | 数時間〜数日先の需要予測（こちらは別途バッチモデルが必要）   |

この区別は重要で、**用途によって参照すべき指標が変わる**。リアルタイム配車支援なら 1h ラグ版、翌日のシフト計画なら 24h ラグ版（あるいはラグなしモデル）を採用すべきである。

### 5.2 「外生変数なし」の限界

本モデルは Colab 無料枠で動かす制約上、**気象・祝日・イベント情報を一切使っていない**。そのため次のようなケースで予測を外す可能性が高い:

- 突発的な大雨・降雪（タクシー需要は急増する）
- ヤンキースタジアム／マディソンスクエアガーデンのイベント終了時刻
- 大型連休・観光シーズン

実運用時にはこれらの**外生変数を追加投入する余地**があり、`bam` の枠組みのまま `s(temp) + s(precip) + holiday_flag` を追加するだけで拡張できる（GAM の加法性の強み）。

### 5.3 「Negative Binomial を選ぶ」という地味だが重要な判断

需要データは整数のカウントで、メッシュごとに分散がきわめて大きい（深夜の住宅地メッシュは 0 が連続し、ミッドタウンの夕方は 100 を超える）。これに Poisson を当てると分散を過小評価し、**信頼区間が異常に狭く出てしまう**。負の二項分布を採用したことで、過分散下でも信頼区間が現実的な幅に保たれ、後段の意思決定（「ここに 1 台配備すべきか」）が安全側に倒れる。

### 5.4 「循環スプライン (`cc`)」の効果

時間軸を普通の `tp` スプラインで表現すると、23 時の予測値と翌 0 時の予測値が不連続になりうる。`bs = "cc"`（cyclic cubic spline）で 24 時間周期を強制したことで、深夜帯の予測がなめらかにつながり、**業務系の可視化ダッシュボードでも違和感のないアウトプット**になる。地味だが、現場で使うモデルではこうしたディテールが効く。

## 6. 実運用への提案：二系統モデルによる配車計画フロー

このプロジェクトでは、**配車オペレーションの時間軸の違い**に応じて 2 つのモデルを使い分ける運用を提案する。

```
                ┌─────────────────────────────────────────────────────┐
                │            過去の乗車履歴データ                       │
                └─────────────────────────────────────────────────────┘
                          │                              │
                          ▼                              ▼
        ┌──────────────────────────┐    ┌──────────────────────────┐
        │ ① 前日バッチ予測          │    │ ② リアルタイム予測         │
        │   モデル B (24h ラグ)     │    │   モデル A (1h ラグ)       │
        │                          │    │                          │
        │   入力: 24 時間前の需要    │    │   入力: 直前 1 時間の需要  │
        │   出力: 翌日各時間帯の     │    │   出力: 次の 1 時間の       │
        │        メッシュ別需要量    │    │        メッシュ別需要量    │
        │                          │    │                          │
        │   目的: ドライバーシフト   │    │   目的: 空車を需要発生地点  │
        │        車両配置の事前計画  │    │        に誘導 (動的配備)   │
        └──────────────────────────┘    └──────────────────────────┘
                          │                              │
                          ▼                              ▼
        ┌──────────────────────────┐    ┌──────────────────────────┐
        │ 翌日のオペレーション計画書 │    │ ドライバーアプリへ配信      │
        │ (シフト・拠点配置)         │    │ (「次は◯◯エリアへ」)       │
        └──────────────────────────┘    └──────────────────────────┘
```

> 💡 **重要な設計思想**：
> 1 つのモデルですべての時間軸をカバーしようとせず、**「何時間先を予測するか」によってモデルを分ける**。
> リアルタイム予測には直近情報を、バッチ予測にはより長いラグを使うことで、それぞれの用途で最も使える情報を最大限活用する。

## 7. 他都市・他事業者への横展開

本モデルは「乗車の **日時** と **位置情報** さえあれば動く」最小構成になっており、入力データを差し替えるだけで他都市・他事業者に展開できる。

| 差し替えポイント   | 設定箇所                                     |
| ------------------ | -------------------------------------------- |
| 入力 CSV           | `fread("/content/train.csv")` の読み込み列名 |
| 対象エリア         | `lon_seq`, `lat_seq` の範囲指定              |
| メッシュ解像度     | `round(..., 2)` の桁数（1km 単位を変更可）   |
| 外生変数（任意）   | `bam()` の formula に `s(temp) + holiday` 等 |

東京・大阪などの都市データはもちろん、フードデリバリーの注文需要、シェアサイクルのステーション別利用需要など、**「日時 + 位置」の構造を持つあらゆる需要予測**に応用可能である。
